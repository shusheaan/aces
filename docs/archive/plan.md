# 无人机空战仿真项目 — 总体计划

## 1. 项目概述

两架四旋翼无人机在有障碍物的 3D 环境中进行空战对抗（dogfight）。每架无人机只有一个前向摄像头，需要通过第一人称视角感知环境和对手，进行追逐、躲避、锁定。

本项目从纯仿真开始，逐步叠加技术层次，最终目标是覆盖从底层控制到高层策略学习的完整机器人技术栈。

### 1.1 核心设计理念

- **第一人称视角优先**：不使用上帝视角/全知信息，模拟真实机载传感器的限制
- **渐进式复杂度**：每一层独立可交付，不需要一步到位
- **Rust 优先**：核心仿真和控制用 Rust，Python 生态成熟的部分（RL、CV、CasADi）用 Python
- **先自己写，后接框架**：先用自己的仿真器理解原理，后期再接 PX4/ROS2

### 1.2 空战规则

```
每架无人机装备：
  - 1 个前向固定摄像头（FOV = 90°，分辨率 320×240）
  - 摄像头固定在机头方向，随无人机姿态旋转

锁定（Lock-on）条件：
  1. 对手在己方摄像头视场锥（FOV cone）内
  2. 对手距离 ≤ D_lock（可调，默认 2.0 m）
  3. 条件 1 和 2 同时持续满足 T_lock 秒（可调，默认 1.5 s）
  → 判定击杀

出局条件：
  - 被对方锁定击杀
  - 撞击障碍物或地面
  - 飞出边界

对称博弈：双方同时是追击方和逃跑方，谁先锁定对方谁赢。
```

锁定检测的数学：

```
设 A 为判定方，B 为目标方

相对位置:  d = pos_B - pos_A
机头朝向:  f_A = R(q_A) · [1, 0, 0]     （R 为姿态旋转矩阵）

夹角:      θ = arccos( (d · f_A) / ||d|| )

锁定条件:  θ < FOV/2  且  ||d|| < D_lock
持续计时:  lock_timer += dt  如果条件持续满足
           lock_timer = 0    如果条件中断
击杀判定:  lock_timer ≥ T_lock
```

### 3.1 四旋翼动力学模型

#### 状态空间（13 维，四元数表示）

```
x = [px, py, pz,         -- 位置 (世界坐标系, 3)
     vx, vy, vz,         -- 速度 (世界坐标系, 3)
     qw, qx, qy, qz,    -- 姿态四元数 (4, ||q||=1)
     wx, wy, wz]         -- 角速度 (机体坐标系, 3)
```

选择四元数而非欧拉角的原因：
- 没有万向锁（gimbal lock），大角度机动时不会奇异
- 狗斗可能出现大角度姿态变化
- 四元数乘法比旋转矩阵计算量小

#### 控制输入（4 维）

```
u = [f1, f2, f3, f4]     -- 四个电机推力
约束: 0 ≤ fi ≤ f_max
```

#### 动力学方程

```
位置导数:
  ṗ = v

速度导数 (世界坐标系):
  v̇ = [0, 0, -g] + (1/m) · R(q) · [0, 0, F_total] + (1/m) · F_drag
  其中:
    F_total = f1 + f2 + f3 + f4
    R(q) = 四元数 q 对应的旋转矩阵
    F_drag = -k_drag · v  (线性空气阻力，简化模型)

四元数导数:
  q̇ = 0.5 · q ⊗ [0, wx, wy, wz]
  其中 ⊗ 为四元数乘法
  每步后需要归一化: q ← q / ||q||

角速度导数 (机体坐标系):
  I · ẇ = τ - ω × (I · ω)
  其中:
    I = diag(Ixx, Iyy, Izz)  惯性矩阵（对角化简化）
    τ = M · u                 混合矩阵将电机力映射到力矩
    ω × (Iω) 为陀螺效应项

混合矩阵 M (X 型布局):
  ┌                                        ┐   ┌    ┐     ┌        ┐
  │  1          1          1          1    │   │ f1 │     │ F_total│
  │  d/√2     -d/√2     -d/√2      d/√2  │ · │ f2 │  =  │ τ_x    │
  │  d/√2      d/√2     -d/√2     -d/√2  │   │ f3 │     │ τ_y    │
  │  c_τ      -c_τ       c_τ      -c_τ   │   │ f4 │     │ τ_z    │
  └                                        ┘   └    ┘     └        ┘

  d = 臂长（电机到重心的距离）
  c_τ = 扭矩系数（电机反扭矩与推力的比值）
```

#### Crazyflie 2.1 物理参数（参考值，需核实）

```
质量:        m    = 0.027 kg
臂长:        d    = 0.04 m
惯性矩:     Ixx  = 1.4e-5 kg·m²
             Iyy  = 1.4e-5 kg·m²
             Izz  = 2.17e-5 kg·m²
最大单电机推力: f_max ≈ 0.15 N  (总推力 ≈ 0.6N, 推重比 ≈ 2.2:1)
扭矩系数:    c_τ  = 0.005964
阻力系数:    k_drag ≈ 0.01
重力:        g    = 9.81 m/s²
悬停单电机推力: f_hover = mg/4 ≈ 0.066 N
```

注意：Crazyflie 推重比约 2.2:1，机动能力有限。仿真中可以适当增大推力上限来获得更激烈的狗斗效果，但要标注这不是真实参数。

#### 数值积分

使用 4 阶 Runge-Kutta (RK4)：

```
仿真内步长: dt_sim = 0.001 s  (1000 Hz)
控制频率:   dt_ctrl = 0.01 s  (100 Hz)
每个控制周期内做 10 步 RK4 积分
```

#### 验证方法

1. 给悬停推力 fi = mg/4，无人机应静止不动（误差 < 1e-10）
2. 关闭电机，自由落体加速度应为 g
3. 给不对称推力，应产生对应方向的旋转
4. 能量守恒：无阻力时，动能 + 势能 + 电机做功应平衡

### 3.2 3D 环境

#### 环境表示

使用 Signed Distance Field (SDF)：

```
SDF(p) = 到最近障碍物表面的有符号距离
  > 0  自由空间
  = 0  表面
  < 0  障碍物内部

对于简单几何体 (方块、圆柱、球)，SDF 有解析公式：

方块 SDF (中心 c, 半尺寸 h):
  d = |p - c| - h              (逐分量绝对值和减法)
  SDF = ||max(d, 0)|| + min(max(dx, dy, dz), 0)

多个物体: SDF_total(p) = min(SDF_1(p), SDF_2(p), ...)

地面: SDF_ground(p) = pz

天花板: SDF_ceiling(p) = H - pz

边界墙: 类似处理
```

#### 碰撞检测

```
每一步检查: SDF(p_drone) < r_drone  →  碰撞  →  出局
r_drone ≈ 0.05 m (Crazyflie 等效碰撞半径)
```

#### 默认场景："仓库"

```
空间: 10m × 10m × 3m

障碍物布局 (俯视图):
  ┌──────────────────────────┐
  │                          │
  │    ██          ██        │
  │    ██          ██        │
  │                          │
  │         ██               │
  │         ██               │
  │                          │
  │    ██          ██        │

  └──────────────────────────┘
```

### 3.3 MPPI 控制器

#### 追击代价函数

```
stage_cost_pursuer(x_self, x_enemy, u) =
    w_lock · lock_reward(x_self, x_enemy)     -- 锁定奖励(负代价)
  + w_dist · ||p_self - p_enemy||²             -- 接近对手
  + w_face · (1 - cos(angle_to_enemy))         -- 面朝对手
  + w_vel  · ||v||² · 𝟙{||v|| > v_max}        -- 超速惩罚
  + w_ctrl · ||u - u_hover||²                  -- 控制平滑
  + w_obs  · obstacle_cost(p_self)             -- 避障

obstacle_cost(p) =
    0                          如果 SDF(p) > d_safe
    (d_safe - SDF(p))²         如果 0 < SDF(p) ≤ d_safe
    极大值 (1e6)               如果 SDF(p) ≤ 0

lock_reward(x_self, x_enemy):
    d = p_enemy - p_self
    θ = arccos(d · forward_self / ||d||)
    如果 θ < FOV/2 且 ||d|| < D_lock:
      return -w_bonus · (1 - θ/(FOV/2)) · (1 - ||d||/D_lock)
    否则:
      return 0
```

#### 逃跑代价函数

```
stage_cost_evader(x_self, x_enemy, u) =
    w_escape · escape_cost(x_self, x_enemy)    -- 脱离对手视场
  + w_dist  · max(0, D_safe - ||p_self - p_enemy||)²  -- 保持距离
  + w_ctrl  · ||u - u_hover||²
  + w_obs   · obstacle_cost(p_self)

escape_cost(x_self, x_enemy):
    -- 计算自己是否在对手的视场内
    d = p_self - p_enemy
    θ_enemy = arccos(d · forward_enemy / ||d||)
    如果自己在对手 FOV 内且距离 < D_lock:
      return (1 - θ_enemy/(FOV/2))              -- 越正对越危险
    否则:
      return 0  -- 安全
```

## 4. Level 1 — 策略学习 (RL)

### 4.0 目标

- 用强化学习训练追逐和逃跑策略
- 对比 RL vs MPPI 的表现
- 自我博弈 (self-play) 让策略自动进化

### 4.1 RL 环境定义 (Gymnasium 接口)

```python
class DroneDogfightEnv(gymnasium.Env):
    """
    通过 pyo3 调用 Rust 仿真核心
    """

    # 观测空间 (己方视角, 相对坐标)
    observation_space = Box(shape=(18,))
    # [己方速度(3), 己方角速度(3),
    #  对手相对位置(3), 对手相对速度(3),
    #  己方姿态(roll,pitch,yaw)(3),
    #  到最近障碍物距离(1), 锁定计时器(1), 被锁定计时器(1)]

    # 动作空间 (期望推力增量, 连续)
    action_space = Box(shape=(4,), low=-1, high=1)
    # 映射到实际推力: u = u_hover + action * δ_max

    def step(self, action):
        # 调用 Rust 仿真核心前进一步
        obs, reward, done, truncated, info = self.sim.step(action)
        return obs, reward, done, truncated, info

    def reset(self):
        # 随机初始位置 (双方在场地两端, 加随机偏移)
        return self.sim.reset()
```

### 4.2 奖励函数设计

```
reward_pursuer(t) =
  + 100.0           如果成功锁定对手 (终局奖励)
  - 100.0           如果被对手锁定
  - 50.0            如果撞墙
  + 0.1 · Δ_lock    锁定进度增量 (每步的小奖励)
  - 0.01 · ||u||²   控制量惩罚
  - 0.05 · Δ_dist   如果离对手变远了 (鼓励接近)

reward_evader(t) =
  + 100.0           如果成功锁定对手 (逃跑方也能反杀)
  - 100.0           如果被对手锁定
  - 50.0            如果撞墙
  + 0.1 · Δ_escape  脱离对手 FOV 的奖励
  + 0.01 · dt       存活奖励 (活得越久越好)
```

注意：奖励设计是 RL 最难的部分。上面是初始版本，大概率需要反复调整。如果 agent 学到了钻空子的行为（比如一直躲在角落不动），需要修改奖励。

### 4.3 训练方案

```
方案 A: 先分别训练，再对打
  1. 训练 pursuer: 对手用固定策略 (直线飞行)      → PPO, ~1M steps
  2. 训练 evader: 对手用训练好的 pursuer           → PPO, ~1M steps
  3. 对打: trained pursuer vs trained evader

方案 B: 自我博弈 (Self-Play)
  1. 初始化两个相同的策略网络
  2. 双方互为对手训练
  3. 定期保存历史版本，从历史版本池中随机抽对手
  4. 避免策略循环 (A 克 B, B 克 C, C 克 A)

推荐先做方案 A (简单, 能快速看到结果), 再做方案 B。
```

### 4.4 网络结构

```
策略网络 (Actor):
  Input (18) → Linear(128) → ReLU → Linear(128) → ReLU → Linear(4) → Tanh
  输出: [-1, 1] 的连续动作

价值网络 (Critic):
  Input (18) → Linear(128) → ReLU → Linear(128) → ReLU → Linear(1)
  输出: 状态价值估计

注意：这是一个简单的 MLP。不需要 CNN 因为 Level 1 还没有视觉输入。
```

### 4.5 对比实验

| 对战组合 | 目的 |
|---------|------|

## 5. Level 2 — 不确定性处理

### 5.0 目标

- 加入观测噪声和风扰动
- 实现状态估计 (EKF / 粒子滤波)
- 实现 Chance-Constrained MPC / Risk-Aware MPPI

### 5.1 噪声模型

#### 位置观测噪声

```
仿真中对手的真实位置: p_true
实际观测到的位置:    p_obs = p_true + ε,  ε ~ N(0, σ_obs² · I)
σ_obs = 0.1 m (可调)

含义: 你知道对手大概在哪，但有 ~10cm 的误差
```

#### 风扰动模型

```
Ornstein-Uhlenbeck 过程 (有均值回复的随机风):

  dw = θ_w · (μ_w - w) · dt + σ_w · dW

  θ_w = 2.0     (回复速率, 越大风变化越快)
  μ_w = [0,0,0]  (平均风速, 可设为非零模拟持续侧风)
  σ_w = 0.3 N    (风力波动强度)

在速度方程中加入: v̇ += w(t) / m

含义: 持续的随机推力扰动，无人机需要主动抵抗
```

### 5.2 状态估计

#### 扩展卡尔曼滤波 (EKF)

```
用于: 融合 IMU + 位置观测，估计对手的状态

状态: x_enemy = [px, py, pz, vx, vy, vz]  (6 维, 只估计位置和速度)

预测步:
  x̂⁻ = F · x̂ + B · û     (匀速模型: 假设对手匀速运动)
  P⁻ = F · P · Fᵀ + Q     (Q 为过程噪声, 反映对手可能的加速)

更新步 (收到新观测时):
  K = P⁻ · Hᵀ · (H · P⁻ · Hᵀ + R)⁻¹
  x̂ = x̂⁻ + K · (z - H · x̂⁻)
  P = (I - K · H) · P⁻
```

#### 粒子滤波

```
用于: 非高斯分布的情况 (比如对手可能在墙左边也可能在墙右边)

N_particles = 500
每个粒子: 一个假设的对手状态

预测: 每个粒子按运动模型 + 噪声前进
更新: 根据观测似然重新加权
重采样: 低权重粒子被高权重粒子替代

比 EKF 计算量大, 但能处理多模态分布
```

### 5.3 Chance-Constrained MPPI

```
标准 MPPI: 在确定性动力学下采样轨迹
Risk-Aware MPPI: 每条轨迹加入不同的风实现

修改 rollout:
  for k in 1..K:
    x = x₀
    w_traj = sample_wind_trajectory()  ← 采样一条风的时间序列
    for t in 0..N-1:
      u_t = U[t] + δU⁽ᵏ⁾[t]
      x = dynamics(x, u_t) + wind_effect(w_traj[t])  ← 加入风
      S⁽���⁾ += stage_cost(x, u_t)

风险过滤:
  对每条轨迹计算碰撞风险:
    risk⁽ᵏ⁾ = max_t (r_drone - SDF(p_t⁽ᵏ⁾))  (正值 = 穿透障碍物)
  
  方法 A — 硬过滤:
    w⁽���⁾ = 0  如果 risk⁽ᵏ⁾ > 0   (丢弃碰撞轨迹)

  方法 B — CVaR 加权:
    对 risk 排序, 取最差的 α% (比如 5%)
    这些轨迹的代价额外加一个大的惩罚项
    
  方法 B 对应金融中的 CVaR 约束:
    "在最差 5% 的情况下，碰撞风险仍可控"
```

### 5.4 CasADi/ACADOS Chance-Constrained MPC

```
方法: Scenario-based approach

在 MPC 中, 采样 M 个风扰动场景 w⁽¹⁾, ..., w⁽ᴹ⁾:

  min  Σ_t cost(x_t, u_t)
  s.t. x_{t+1}⁽ⁱ⁾ = f(x_t⁽ⁱ⁾, u_t, w_t⁽ⁱ⁾)   ∀ i=1..M (每个场景)
       SDF(p_t⁽ⁱ⁾) ≥ d_safe                       ∀ i, t   (所有场景都安全)
       u_min ≤ u_t ≤ u_max

注意: 控制量 u_t 对所有场景相同 (这里是开环鲁棒)
     更高级的做法是 affine disturbance feedback: u_t = ū_t + K · (x_t - x̄_t)

M 通常取 10-50, 太多 NLP 太大解不动
```

## 6. Level 3 — 视线遮挡 + 信息不对称

### 6.0 目标

- 实现视线遮挡（被障碍物挡住就看不到对手）
- 看不到对手时，用预测模型估计对手位置
- 信念状态维护 (粒子滤波)
- 行为预测模型

### 6.1 视线检测 (Ray Casting)

```
A 能否看到 B？

从 pos_A 到 pos_B 射一条线段。
沿线段均匀采样点 (或用 sphere tracing):

  ray(t) = pos_A + t · (pos_B - pos_A),  t ∈ [0, 1]

  Sphere tracing (利用 SDF 加速):
    t = 0, p = pos_A
    while t < 1:
      d = SDF(p)
      if d < ε:           → 射线碰到障碍物 → 看不见
        return false
      t += d / ||pos_B - pos_A||   ← SDF 保证这步内不会穿过障碍物
      p = ray(t)
    return true            → 射线未被阻挡 → 看得见
```

Sphere tracing 比均匀采样快很多（SDF 值大时可以跨大步），而且你已经有 SDF 了。

### 6.2 信息不对称下的状态维护

```
三种状态:
  1. 可见: 直接观测对手位置 (+ 噪声)
  2. 刚丢失: 知道对手最后位置, 不确定性随时间增长
  3. 长时间丢失: 对手可能在任何合理位置

用粒子滤波维护信念:
  - 可见时: 粒子聚集在观测位置附近 (低不确定性)
  - 丢失后: 粒子按运动模型扩散 (不确定性增长)
  - 重新看到时: 粒子重新聚集 (不确定性骤降)

  粒子运动模型 (对手预测):
    p_{t+1} = p_t + v_t · dt + ε_process
    v_{t+1} = v_t + ε_accel
    
    + 约束: 粒子不能在障碍物内 (SDF > 0)
    + 约束: 粒子不能飞出边界
```

### 6.3 行为预测

```
简单版: 匀速/匀加速预测
  对手未来位置 ≈ p + v·t + 0.5·a·t²  (用最后已知的速度和加速度)

进阶版: 学习对手策略
  输入: 对手最近 T 步的状态序列 [x_{t-T}, ..., x_{t}]
  输出: 对手未来 K 步的预测轨迹 [x̂_{t+1}, ..., x̂_{t+K}]
  
  模型选择:
    - LSTM: 简单有效, 适合序列预测
    - Transformer: 更强但需要更多数据
    - 对两架无人机来说 LSTM 就够了, Transformer 在多机时更有优势

  训练数据: 从 Level 1 的 RL 自我博弈中收集对局��迹
```

### 6.4 基于信念的决策 (POMDP 近似)

```
标准 MPPI: 已知对手准确位置 → 一条轨迹
信念 MPPI: 对手位置是分布 → 多条可能的轨迹

做法:
  从粒子滤波的粒子中采样 M 个对手位置假设
  对每个假设, 跑 MPPI
  取所有假设下代价最低的控制 (minimax)
  或取加权平均 (期望最优)

  这本质上是 robust optimization:
    minimax → worst-case
    期望最优 → risk-neutral
    CVaR → risk-averse
    
  又回到你的金融优化框架了。
```

## 7. Level 4 — 第一人称视觉

### 7.0 目标

- 不再给坐标，完全从摄像头图像感知
- 目标检测：从图像中找到对手
- 深度感知：估计距离和环境结构
- 实时环境建图

### 7.1 仿真摄像头

```
使用 MuJoCo 或 PyBullet 的渲染功能:

每帧输出:
  - RGB 图像: 320 × 240 × 3
  - 深度图:   320 × 240     (每个像素到摄像头的距离)

摄像头参数:
  FOV:         90°
  安装位置:    无人机重心前方
  安装方向:    沿机头方向 (随姿态旋转)
  帧率:        30 fps (每 3 个控制周期出一帧)
```

### 7.2 目标检测 (对手识别)

```
输入: RGB 图像 (320×240)
输出: 对手 bounding box + 置信度

方法:
  - YOLOv8-nano 或类似的轻量级检测器
  - 训练数据: 从仿真中自动生成 (知道对手 3D 位置, 投影到图像坐标)
  - 标注自动化: 仿真的好处是 ground truth 免费
  
从 bounding box 恢复 3D 信息:
  - 方位角: 从 bbox 中心在图像中的位置推算
  - 距离: 从 bbox 大小推算 (已知对手物理尺寸)
        或直接读深度图中 bbox 区域的深度
  - 精度不如直接给坐标, 但更接近真实
```

### 7.3 环境建图

```
每帧的深度图 → 局部点云:
  对每个像素 (u, v):
    depth = depth_map[u, v]
    p_camera = depth · back_project(u, v, FOV, resolution)
    p_world = R(q_drone) · p_camera + p_drone

点云积累 → 占据栅格 (Occupancy Grid):
  3D 体素网格, 每个体素标记为:
    占据 (有障碍物)
    空闲 (确认没有障碍物)
    未知 (没探索过)

  更新规则 (log-odds):
    l(m|z) = l(m) + log(p(z|m)/p(z|¬m))

  或直接用 SDF 更新:
    逐步用观测到的点云更新一个 TSDF (Truncated SDF)

注意: 这部分计算量大。对于两架小无人机在仓库场景中,
如果体素分辨率 10cm, 10m×10m×3m = 30000 个体素, 还算可控。
```

### 7.4 全视觉闭环架构

```
摄像头图像 (320×240 RGB + 深度)
        ↓
┌─────────────────────┐
│ 目标检测 (YOLOv8)   │ → 对手方位 + 距离估计
└─────────────────────┘
        ↓
┌─────────────────────┐
│ 深度 → 点云 → 建图   │ → 局部占据栅格/TSDF
└─────────────────────┘
        ↓
┌─────────────────────┐
│ 状态估计 (EKF)       │ → 对手位置估计 (融合检测结果)
│ + 粒子滤波           │ → 信念状态 (看不见对手时)
└─────────────────────┘
        ↓
┌─────────────────────┐
│ 行为预测 (LSTM)      │ → 对手未来轨迹预测
└─────────────────────┘
        ↓
┌─────────────────────┐
│ MPPI / MPC / RL      │ → 控制指令
└─────────────────────┘
        ↓
    电机推力 → 仿真环境
```

```
  /drone/cmd_thrust        → MAVROS       → PX4
```

### 8.4 Level 5 实现步骤

```
步骤 5.1: 选择硬件方案
  - 根据预算和目标决定

步骤 5.2: 单机悬停
  - 先让一架飞起来并稳定悬停
  - 这一步就会暴露很多 sim-to-real 问题

步骤 5.3: 单机轨迹跟踪
  - 让一架按仿真中生成的轨迹飞
  - 调参使跟踪误差可接受

步骤 5.4: System Identification
  - 用真机飞行数据反推动力学参数
  - 更新仿真模型

步骤 5.5: 双机对抗
  - 从简单场景开始 (开阔空间, 低速)
  - 逐步加入障碍物和提高速度
```

## 9. 合理性反思

### 9.1 这个项目的真实难度

| Level | 估计难度 | 最可能卡住的地方 |
|-------|---------|----------------|
| 0 | 中等 | MPPI 调参（代价函数权重）、可视化搭建 |
| 1 | 中高 | RL 奖励设计、训练不收敛、pyo3 桥接 |
| 2 | 中等 | EKF 调参（Q/R矩阵）、风模型标定 |
| 3 | 高 | 粒子滤波效率、信念-MPPI 计算量、行为预测数据量 |
| 4 | 很高 | 渲染性能、检测模型训练、全视觉闭环调试 |
| 5 | 极高 | 硬件调试、sim-to-real gap、安全问题 |

### 9.2 我之前可能高估的

- **MPPI 性能 "< 10ms"**：1024 采样 × 50 步在 Rust + rayon 下更可能是 10-30ms，仍然实时但不是那么轻松
- **ACADOS 上手**：API 文档不友好，第一次配通可能要几天
- **RL 训练**："~1M steps" 是乐观估计，奖励设计不好的话可能需要反复迭代
- **嵌入式部署 (Level 5)**：比仿真的工作量大很多，不是"选做"那么轻松

### 9.3 我之前可能低估的

- **代价函数/奖励函数调参**：这会占每个 Level 大量时间，没有公式能告诉你正确的权重
- **可视化的重要性**：好的可视化能加速调试 10 倍，值得在 Level 0 多投入时间
- **两架无人机同时仿真的复杂度**：通信、同步、两套控制器并行运行，比单机复杂不止两倍

### 9.4 风险和退出条件

- 如果 Level 0 的 MPPI 调不出有意义的追逃行为 → 先简化成 2D 平面追逃
- 如果 RL 训练不收敛 → 先用 imitation learning（模仿 MPPI 的行为）
- 如果 Level 4 视觉太慢 → 降低分辨率或帧率，或退回 Level 3 的坐标 + 遮挡
- 如果某个 Level 做不下去 → 每个 Level 独立可交付，停在任何一层都有成果

### 9.5 这个项目已经存在的开源工作

需要在开始前调研清楚，避免重复造轮子：

- `gym-pybullet-drones`: 四旋翼 RL 环境，可以参考环境设计
- `lsy_drone_racing`: ETH Zurich 的无人机竞速环境
- `crazyswarm2`: Crazyflie 多机编队的 ROS2 包
- DARPA AlphaDogfight 的公开资料（固定翼，但对抗逻辑可参考）

建议：先看这些项目的代码和设计，借鉴好的部分，在此基础上做你自己的版本。不需要从零发明一切。
